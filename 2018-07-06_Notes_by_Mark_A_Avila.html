Internet Applications and Network Programming

Topic 1: Internet Services

  • The Internet offers diverse services such as web browsing, text messaging, and video streaming
  • None of these services are part of the underlying communications infrastructure of the Internet
  • The Internet provides a general communications mechanism on which all services are built on.
  • Individual services are supplied by application programs that run on computers attached to the Internet.
  • It's possible to devise new services without changing the Internet.

Topic 2: Two Basic Internet Communication Paradigm

Stream Paradigm Message Paradigm
Connected-oriented Connectionless
1-to-1 communication Communication may involve many other parties
Send sequences of individual bytes Send sequences of discrete messages
Arbitrary length transfer Each message limited to 64 KB
Used by most applications Used for multimedia applications
Runs over “Transmission Control Protocol” (TCP) Runs over “User Datagram Protocol” (UDP)

Topic 3: Stream Transport Paradigm

  • A sequence of bytes flows from one application to another
  • Internet arranges for two streams between a pair of communicating applications, one in each direction.
  • The browser uses the stream service to communicate with a web server
  • The browser sends a request, then the web server responds by sending a response.
  • The Internet accepts data flowing from each of the 2 applications, then delivers the data to the other application
  • The stream is sent without attaching meaning to the bytes and without inserting boundaries
  • The sending application can send one byte at a time, or can generate large blocks of bytes
  • TThe stream service moves bytes across the Internet and delivers them as they arrive
  • Smaller chunks of bytes might be combined into a large block, or a large block might be divided into smaller chunks

Topic 4: Message Transport Paradigm

  • The network accepts and delivers messages.
  • Each message delivered to a receiver corresponds to a message transmitted by a sender.
  • If there are K bytes in an outgoing (sent) message, then there are K bytes in an incoming (received) message.
  • The message may be sent directly to another computer, then the message may also be broadcast to all computers on a network.
  • Many computers may send messages to a single recipient application. Therefore, the message paradigm can be 1-to-1, 1-to-many, or many-to-1.

It doesn't make any guarantees about whether a message will arrive/the order in which messages are delivered. Messages may be:

  • Lost (never delivered)
  • Duplicated (more than one copy arrives)
  • Delayed (some packets may take a long time to arrive)
  • Delivered out of order

An application programmer choosing the message paradigm must ensure that the application operates correctly, even if packets are lost or reordered.

Topic 5: What will it be: Stream or Message?

Because The Message Paradigm requires this expertise in design of protocols - to handle lost, duplicated, delayed or reordered - less than 5% of packets in the Internet uses the Message Paradigm. Exceptions might include:

  • Situations where broadcast is needed
  • Where a receiver must play the data as it is received - audio phone call

Our focus will therefore be on Stream Paradigm

Topic 6: Connection-Oriented Communication

  • CREATE CONNECTION: Before the 2 applications can communicate, they must request that a connection is to be created between them
  • EXCHANGE DATA: Once the connection is established, data can be sent in either direction
  • CLOSE CONNECTION: When the data transmission (sending and receiving) is done, the connection can be terminated.

Topic 8: Client Software vs Server Software

Client Software Server Software
Arbitrary application that becomes a client temporarily when remote access is needed Consists of special-purpose, privileged program dedicated to providing a service
Is invoked directly by a user, and executes only for one session Is invoked automatically when a system boots, and continues to execute through many sessions
Runs locally on a user’s computer or device Runs on a dedicated computer system
Actively initiates contact with a server Waits passively for contact from arbitrary remote clients
Can access multiple services as needed, but usually contacts one remote server at a time Can accept connections from many clients at the same time, but usually offers one service
Does not require especially powerful hardware Requires powerful hardware and a sophisticated operating system

Topic 9: Multiple Clients and Multiple Servers

Any given computer can run:

  • A single client
  • A single server
  • Multiple copies of a client that contact a given server
  • Multiple pages each connecting to the same retailer, comparing products
  • Multiple different clients that each contact a particular server
  • Ex. web browser, instant message application, video teleconference
  • Multiple servers, each for a particular service
  • Shares the overhead of the resource
  • Demand on service may be sporadic, server sits idle

    • Topic 10: Server Identification

    How does the client identify the server it needs? Well, the:

  • Identifier that specifies the computer on which the server runs
  • Provide an Internet Protocol (IP) address in IPv4 or IPv6 format
  • Alternatively and probably more commonly - provide a domain name which we look up in the Domain Name System (DNS)
  • Identifier that specifies the particular service on the computer
  • Each service is assigned to a unique 16-bit identifier known as a protocol port number (or “port”).
  • For example; an email is port 25,whiel the WWW is port 80
  • When server begins execution, it registers with its local system by specify the port it will be using
  • Can have conflict with “port already in use”

    Topic 12: Peer-to-Peer (P2P) Architecture

    A bottleneck can occur if a single server provides a given service and multiple clients try to access it. Can the services be provided without creating a central bottleneck?

  • Use file-sharing application, Peer-to-Peer (P2P) Architecture for:
  • Each client is now also functioning as a server
  • Data is distributed (roughly) equally among N servers
  • Reduces traffic between any given server and the Internet to 1/N
  • For large N, this can considerably reduce traffic
